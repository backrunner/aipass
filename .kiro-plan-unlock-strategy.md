# 1Password-style unlock strategy

## Problem

Users get locked out far more often than 1Password. Root causes found in the code:

1. **Idle is measured from AIPass requests, not system activity.** `lock_if_idle`
   (`crates/aipass-agent/src/session.rs:269`) compares `now - last_activity_at`, and
   `last_activity_at` only advances when a request reaches the agent. So if you're
   active in the browser/editor but not triggering an AIPass lookup, the 15-min timer
   still expires and locks. 1Password instead locks only after *system-wide* keyboard/
   mouse idle.
2. **`lock_on_sleep` / `lock_on_screen_lock` are Windows-only.** They're wired in
   `windows_service.rs` but there is **no macOS listener** — on macOS these flags do
   nothing, so sleeping the Mac doesn't lock, and waking it doesn't matter either.
3. **Every agent restart forces re-unlock.** The session key lives only in memory
   (`SessionState::Unlocked`), and startup always begins `Locked` with
   `LockReason::AgentRestart` (`server.rs:74`). A reboot, logout, crash, or app update
   means re-typing the password.
4. **Default idle is 15 min**, shorter than typical.

Decision: ship the new behavior as the **default** (changes existing + new vaults), covering all four points.

## Design (mirrors 1Password)

### A. System-wide idle (macOS) — `idle_lock_minutes`
- Add `crates/aipass-agent` dep `objc2-core-graphics` and read OS HID idle via
  `CGEventSourceSecondsSinceLastEventType(HIDSystemState, AnyInputEventType)`.
- New `session::system_idle_seconds() -> Option<f64>` (macOS impl; `None` elsewhere).
- In `lock_if_idle`: lock when **both** the request-idle AND system-idle exceed the
  threshold. `None` (non-macOS) falls back to request-idle only — current behavior.
  This is the core fix: active-anywhere keeps the vault unlocked.

### B. Lock on sleep / screen lock (macOS)
- New `session::spawn_macos_power_watcher(state)` spawned from `run_server_with_state`
  alongside the idle watcher (`server.rs:97`), guarded `#[cfg(target_os = "macos")]`.
- Observe `NSWorkspace` notifications (`willSleepNotification`) and the distributed
  `com.apple.screenIsLocked` notification via `objc2-foundation`; on fire, call
  `lock_session(state, LockReason::SystemSleep | ScreenLock)` honoring the policy flags.
- Reuses the existing `LockReason::SystemSleep` / `ScreenLock` variants — no protocol change.

### C. Persist unlock across agent restarts (Keychain, opt-out via policy)
- 1Password's "Unlock with device": gate a stored secret behind the OS login/biometric.
  The vault can only be re-opened with the password (`Vault::open` is the only path),
  so seal the **password** in the macOS Keychain via the existing `device_secrets`
  module (already used for WebDAV creds, service `dev.aipass.agent`).
- On successful `unlock_with_password` / `create_vault` / `recover_vault`: if policy
  `persist_unlock` is on, store the password under purpose `session.unlock`.
- On agent startup (`run_server`): attempt to read the sealed password and
  `Vault::open`; on success start `Unlocked` instead of `Locked` (still subject to the
  idle/sleep watchers). On failure stay locked.
- On any lock whose reason is idle/sleep/screen-lock/manual, **clear** the sealed
  password so a fresh agent start respects the lock. (Restart is the only reason that
  should auto-reopen.)
- Non-macOS: `device_secrets` set/get are already no-ops → falls back to current
  always-lock-on-restart behavior. No Linux/Windows regression.
- Security note: this trades some at-rest protection (password recoverable by anyone
  with the unlocked OS login keychain) for fewer prompts — exactly 1Password's
  "Unlock with device" tradeoff. It's policy-gated and on by default per the request.

### D. Defaults
- `SessionPolicy::default()` in `aipass-agent-protocol/src/lib.rs`: add
  `persist_unlock: true`, keep `lock_on_sleep`/`lock_on_screen_lock: true`, raise
  `idle_lock_minutes` 15 → 30.
- Mirror in `apps/desktop/src-tauri/src/models.rs` `AppPreferences::default()`
  (`auto_lock_minutes` 15 → 30, add `persist_unlock`).

## Open risk
The persistent-unlock (C) is the security-sensitive part. It is policy-gated and
defaulted on per your instruction; if you'd rather default it **off** (matching
1Password, which ships "Unlock with device" but requires first enabling it), that's a
one-line default change — flag this when reviewing.
